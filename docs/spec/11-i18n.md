# 11 -- Internationalization (i18n)

Add multi-language support to IAgent. Initial target: **English (en, default)** + **Chinese (zh)**. All user-facing strings must be localizable. Developer-facing logs and internal error codes are excluded.

## 1. Existing Infrastructure

| Layer | Tech | Status |
|-------|------|--------|
| Web frontend | `i18next` + `react-i18next` + `i18next-browser-languagedetector` | Active. `en.json` has 17 namespaces, ~150 keys. Wired in `main.tsx`. |
| Go gateway | None | All `writeError()` calls return raw English strings. |
| Python device | None | CLI output uses raw English strings. |
| Python agent | None | Progress result messages are raw English strings. |

### 1.1 Existing i18n Config

File: `web/src/i18n/index.ts`

- Resources: `{ en: { translation: en } }`
- Fallback language: `"en"`
- Detection order: `["navigator", "htmlTag"]`
- Namespaces: flat (no namespace splitting in init; uses `t()` with dot-path keys like `"auth.signIn"`)
- Key format: dot-separated prefix (namespace + key), e.g. `"dashboard.title"`, `"jobs.command"`, `"errors.somethingWentWrong"`

### 1.2 Existing Translation File

File: `web/src/i18n/en.json` (358 lines, 17 sections)

## 2. Language Scope

| Priority | Language | Code | Scope |
|----------|----------|------|-------|
| 1 (current) | English | `en` | All modules, full coverage |
| 2 (new) | Simplified Chinese | `zh` | All modules, full coverage |

Language code format: ISO 639-1 two-letter codes (`en`, `zh`). Follow i18next conventions.

### 4.1 Go i18n Strategy

The frontend cannot directly translate Go error strings (they arrive via JSON). Two approaches:

**Option A (recommended): Error codes + frontend message catalog.**
- Go returns structured errors: `{"error_code": "EMAIL_ALREADY_REGISTERED"}`
- Frontend maps error codes to `t("apiErrors.EMAIL_ALREADY_REGISTERED")` strings
- No Go-side i18n library needed
- Consistent with how the frontend already handles its own i18n

**Option B: Go-side i18n with `Accept-Language` header.**
- Use `golang.org/x/text/message` or a message catalog
- Frontend sends `Accept-Language: zh` header
- Go returns translated strings directly in the response body
- Higher implementation effort; duplicate translation maintenance (Go + JS)

Decision: **Option A**. The frontend already has `i18next` wired. Error codes are stable, language-agnostic, and the frontend can show localized messages without Go changes. Existing raw English strings become the error codes (uppercased, underscored). Write a compat adapter so existing English strings still render for callers that don't use error codes.

### 4.2 Success Messages

Success messages like `"device deleted"`, `"tier updated"` are returned as `{"message": "tier updated"}`. Same treatment: map to message codes (`DEVICE_DELETED`, `TIER_UPDATED`) and translate on the frontend.

## 8. Language Detection & Switching

### 8.1 Frontend

**Current:** `i18next-browser-languagedetector` with order `["navigator", "htmlTag"]`. No persistence (no cookie/localStorage).

**Target:**
- Detection order: `["querystring", "navigator", "htmlTag"]` -- allow `?lang=zh` override
- Add a language switcher component to `SettingsPage` and `Layout` (flag/dropdown)
- Persist choice to `localStorage` (add `"localStorage"` to detection order, after `"querystring"`)
- `<html lang="...">` attribute updates on language change

### 8.2 Gateway

No server-side language detection. Error codes are language-agnostic.

### 8.3 Device CLI

- `--lang zh` CLI flag (all subcommands)
- Fallback: `LANG` / `LC_ALL` env var
- Default: `en`

## 10. Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Go error codes, not Go-side i18n | Frontend already has i18next. Single source of truth for UI strings. Stable codes are API-contract friendly. |
| Flat JSON, dot-separated keys | Already in use. Consistent with existing `en.json`. No namespace splitting in i18next init needed. |
| `zh` not `zh-CN` | Simpler two-letter code. Add `zh-CN` / `zh-TW` later if needed as locale-specific overrides. |
| No backend translation for machine-readable messages | Status codes, error codes, progress codes stay English. Human-readable strings come from locale files. |
| `Accept-Language` not used | Server-side language negotiation adds complexity with no benefit under the error-code approach. |
