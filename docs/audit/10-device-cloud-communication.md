# Audit 04 — Device ↔ Cloud Communication (Result Pull, VNC, Login)

**Date:** 2026-06-09
**Scope:** Inspection of every code path that supports the **return half** of a job's lifecycle and the **interactive surface** between cloud and device. Specifically:

1. Result return path — `JOB_RESULT`, `FILE_PULL_*` (output files) → web UI download
2. Interactive browser — `VNC_OPEN` → device VNC bridge → gateway relay → web `noVNC`
3. Login support — auto-trigger of headed/VNC mode when login is required, plus `CRED_CAPTURE` (save login) and `CRED_PUSH` (re-inject login)

The forward half (`JOB_DISPATCH`, `FILE_PUSH_*`, queueing, allocation) was already covered in audit 02 / 03 and is not re-audited here, except where its bugs feed back into the return path.

Reading order followed: `goal.md` → `00-overview.md` → `01-architecture.md` → `05-tunnel-protocol.md` (especially §4.10, §9, §10) → source code under `gateway/`, `device/`, `agent/`, `web/`.

---

## Summary

| Capability | End-to-end status | Verdict |
|---|---|---|
| Job status / progress events back to web (`JOB_PROGRESS`, `JOB_RESULT`) | Working: device polls agent, emits frames, gateway updates DB and fans out to WS subscribers. | **OK** |
| Result JSON visible on web UI | Working: `GET /jobs/{id}/result` returns `result` field; web shows it and offers a JSON download. | **OK** |
| Output **files** generated by the agent (`/work/output`) returned to user | Frame plumbing exists end-to-end (`FILE_PULL_*`), **but the file path mapping between agent → device → gateway is broken AND there is no download API or UI for output files**. User cannot retrieve any file the agent produces. | **CRITICAL — broken** |
| Pre-staged input files pushed to device on dispatch | `Dependencies.PushFilesToDevice` is a stub returning `nil`. Files are linked to the job in DB and listed in `JOB_DISPATCH.file_ids` but the actual `FILE_PUSH_*` chunks are never emitted. | **CRITICAL — broken** |
| VNC session open: control plane (`POST /jobs/{id}/vnc` → `VNC_OPEN` → device starts agent VNC stack → `VNC_OPENED`) | Wiring is complete on all three hops. | **OK** |
| VNC session data plane: browser noVNC ↔ gateway ↔ device ↔ agent RFB | Pairing works, but cannot actually be used from the web UI today (see "Login support" issues #4 and #5). | **Partial** |
| Auto-trigger VNC when agent encounters a login wall | **Not implemented anywhere.** Agent has no signal/event for "login required" and the device dispatcher / gateway has no auto-open VNC reaction. | **Missing** |
| Save login (`POST /vnc/{id}/save-login` → `CRED_CAPTURE` → vault) | Wiring exists but is broken: `origin` is never collected from the user, so the device asks the agent for an empty origin; capture path is also schema-inconsistent (see #8). | **Broken** |
| Re-inject saved login (`CRED_PUSH` on dispatch) | Working when a customer attaches a `credential_id` to the job. | **OK** |

**Bottom line:** the cloud → device "out" direction works for jobs and progress, but the **return path is in two parts: status/JSON works; file outputs do not**. Live VNC works as a tunnel but the web UI cannot complete the connection because the RFB password and an authenticated WS URL are not returned to the browser. There is no auto-open-on-login-required behaviour anywhere in the stack.

## 1. Result return path — JOB_RESULT, FILE_PULL_*, downloads

### 1.1 `JOB_RESULT` plumbing — works

| Step | Evidence |
|---|---|
| Device dispatcher polls agent `GET /jobs/{id}` and emits `JOB_RESULT` on terminal status | `device/iagent_device/jobs/dispatcher.py:154-174` |
| Gateway `Hub` routes `JOB_RESULT` → `OnJobResult` | `gateway/internal/tunnel/router.go:52-57`, `gateway/internal/tunnel/hub.go:278-281` |
| Wired handler updates job row, publishes `WSEventJobResult`, releases agent | `gateway/cmd/gateway/main.go:164-180` |
| Web UI fetches `GET /api/v1/jobs/{id}/result` and renders `job.result` | `gateway/internal/httpapi/jobs_handler.go:259-284`, `web/src/pages/JobsPage.tsx:270-284` |

This part of the return path is functional.

### 1.2 `FILE_PULL_*` plumbing — frames work, but path mapping is broken

| Step | Evidence | Status |
|---|---|---|
| Device puller scans `cfg.workspace_dir / job_id`, emits `FILE_PULL_BEGIN` / `FILE_PULL_CHUNK` / `FILE_PULL_END` | `device/iagent_device/files/puller.py:22-66` | Code-OK |
| Dispatcher invokes `puller.pull_outputs(job_id)` after `JOB_RESULT` (success only) | `device/iagent_device/jobs/dispatcher.py:164-168` | Code-OK |
| Hub registry routes `FILE_PULL_*` → `FileRelay.OnFilePull*` | `gateway/internal/tunnel/router.go:152-160`, `gateway/cmd/gateway/main.go:304-312` | Code-OK |
| Gateway reassembles, verifies sha256, writes to `{baseDir}/jobs/{job_id}/output/{name}`, sends `FILE_PULL_ACK` | `gateway/internal/relay/relay.go:283-413` | Code-OK |

So the **frame transport** is wired end-to-end. The problems begin at the **filesystem boundaries**.

#### Critical gap C1 — Workspace path mismatch (no output ever appears on the device host)

The dispatcher tells the agent to use:

```python
# device/iagent_device/jobs/dispatcher.py:88
await client.create_job(..., workspace_dir=f"/work/workspaces/{job_id}")
```

The agent honors that and writes outputs to `/work/workspaces/{job_id}/output` (`agent/iagent_agent/runtime/executor.py:139-145`).

But the device only bind-mounts one directory into the container:

```python
# device/iagent_device/docker/manager.py:91, 107
workspace_mount = f"{self.data_dir}/workspaces:/workspaces:rw"
... volumes=[workspace_mount]
```

`/work` is declared as `VOLUME ["/work"]` in `agent/Dockerfile:166`, so Docker creates an **anonymous volume** at runtime that the host has no path to. The bind mount at `/workspaces` (note: **no `/work/`**) is never used by the agent — its path is `/work/workspaces/...`, not `/workspaces/...`.

The puller then does:

```python
# device/iagent_device/files/puller.py:22-23
ws = self.workspace_dir / job_id     # = {data_dir}/workspaces/{job_id} on host
if not ws.exists(): return []
```

This directory is empty. **Every successful job will return zero output files.** The gateway happily handles whatever `FILE_PULL_*` frames it receives, but the device has none to send.

Additionally, the agent container is created with `read_only=True` (`docker/manager.py:105`). Without `/work` being on a writable mount, anything the agent writes outside `/tmp`, `/run`, or the anonymous `/work` volume would fail; and even when it succeeds (because `/work` is a VOLUME) the host cannot see it.

**Fix sketch:** either
- (a) bind-mount the host workspace at `/work/workspaces` (or the whole `/work`) and have the dispatcher continue to pass `/work/workspaces/{job_id}`, or
- (b) keep the bind at `/workspaces` and pass `workspace_dir=/workspaces/{job_id}` to the agent, and update the puller's host path accordingly.

Option (a) is the smaller change.

#### Critical gap C2 — No download endpoint for job output files

`docs/spec/05-tunnel-protocol.md §4.10` says:

> Files are available for download via `GET /jobs/{job_id}/output/{file_id}`.

`gateway/internal/httpapi/router.go:122-130` registers no such route. The gateway saves output files to disk at `{baseDir}/jobs/{job_id}/output/{name}` but exposes no way to retrieve them.

Furthermore `relay.go:386-395` constructs the `model.File` row with `UserID` zero-valued and never inserts it into the `job_files` join table, so even `GET /jobs/{id}` would not surface the relationship between the job and its output files.

#### Significant gap C3 — Web UI does not surface output files

`web/src/pages/JobsPage.tsx:270-284` renders the `result` JSON only and provides a `downloadFile` helper that just stringifies that JSON (`JobsPage.tsx:185-189`). There is no list of attached output files, no download buttons per file, and no API call to such an endpoint (which doesn't exist anyway, see C2).

`web/src/api/schemas.ts` has no `JobOutputFile` type. The `Job` zod schema makes no mention of an output-files array.

#### Significant gap C4 — `PushFilesToDevice` is a stub

```go
// gateway/internal/httpapi/jobs_handler.go:286-289
func (deps *Dependencies) PushFilesToDevice(ctx interface{}, job *model.Job, deviceID model.UUID) error {
    return nil
}
```

When a customer submits a job with `file_ids`, the gateway:

1. Inserts rows into `job_files` (jobs_handler.go:62-65) — OK.
2. Calls `PushFilesToDevice` — no-op.
3. Sends `JOB_DISPATCH` carrying the `file_ids` to the device.
4. The device's stager `_wait_for_files` blocks for 30 s (`dispatcher.py:48-59`) waiting for `FILE_PUSH_*` frames that are never sent, then proceeds anyway.

This is not a return-path bug, but it directly affects the user-visible result: any job that depends on uploaded inputs will run with no inputs in `/work/workspaces/{id}/inputs`. From the user's perspective it appears on the same axis as "result didn't come back" because the result is empty / wrong. Worth flagging in this audit.

#### Minor gap C5 — Output file pull never fires on `failed` jobs

`dispatcher.py:158-174`: `puller.pull_outputs(job_id)` is only called inside the `if status == "succeeded"` branch. If a job fails after producing partial logs / debug artefacts, those are lost. The spec only says "after job completion" (`05 §4.10`); failure is also completion.

#### Minor gap C6 — Puller does not wait for `FILE_PULL_ACK`

`puller.py:68-72` registers an event for each in-flight `file_id` but `_send_file` never `await`s it. The send loop happily goes on to the next chunk / next file regardless of ACKs, so the spec's backpressure rule (≤ 8 chunks in flight, retransmit on ERROR) is not implemented. Compared to the upload path's stager, which is similarly best-effort, this is a mirrored gap; it just becomes more painful when output sizes grow.

## 2. VNC live browser

### 2.1 Control plane — wiring complete

| Step | Evidence |
|---|---|
| Web → `POST /api/v1/jobs/{id}/vnc` | `gateway/internal/httpapi/router.go:133`, `vnc_handler.go:15-67` |
| Gateway creates `vnc_sessions` row + `session_token`, sends `VNC_OPEN` over tunnel | `vnc_handler.go:45-60` |
| Device handles `VNC_OPEN` → calls agent `POST /vnc/start` → emits `VNC_OPENED` with `rfb_password` | `device/iagent_device/vncbridge/bridge.py:36-72` |
| Device dials second WS to `wss://{gateway}/session/{sid}` and bridges TCP RFB ↔ WS | `bridge.py:74-115` |
| Gateway pairs `/session/{sid}` (device) ↔ `/ws/vnc/{sid}` (browser) and pumps bytes | `gateway/internal/vncrelay/relay.go:138-223` |
| `OnVNCOpened` records the RFB password on the session | `gateway/cmd/gateway/main.go:221-227` |

### 2.2 Critical gap V1 — RFB password never reaches the browser

The web UI expects `data.rfb_password` from the `POST /jobs/{id}/vnc` response:

```ts
// web/src/pages/JobsPage.tsx:159-162
openVNC.mutate(effectiveJob.id, {
  onSuccess: (data) => {
    setVncData({ wsUrl: data.ws_url, rfbPassword: data.rfb_password, sessionId: data.session_id });
    ...
```

But the API response struct does not include it:

```go
// gateway/internal/httpapi/vnc_handler.go:62-66
writeJSON(w, http.StatusCreated, model.VNCOpenResponse{
    SessionID: sess.ID,
    WSUrl:     "/ws/vnc/" + sess.ID.String(),
    TTLSecs:   maxTTL,
})
```

`useCredentials.ts:53` and `api/schemas.ts:237-243` declare the field, but the server never populates it. This is structurally impossible from the synchronous POST anyway: the password is only known after the device returns `VNC_OPENED`, which is asynchronous on the tunnel.

`noVNC` will then call the WebSocket and immediately fail RFB authentication. The "Open Browser" button can never produce a working session.

**Fix sketch:** either
- (a) have the POST handler block on a `MarkReady`/`session.Ready()` channel until `VNC_OPENED` lands (with a 10–15 s timeout), then return the password; or
- (b) deliver the password lazily — POST returns 202 + `session_id`, the web client subscribes via the existing `WS /ws` event bus to a `vnc.ready` event carrying the password, then opens noVNC.

Spec §9 step 7 says "Gateway hands rfb_password to the browser (POST response / noVNC config)" — both options are spec-compliant.

### 2.3 Critical gap V2 — Browser noVNC URL has no auth token

```ts
// web/src/components/VNCPanel.tsx:43-46
const rfb = new RFB(canvasRef.current, wsUrl, {
  credentials: { password: rfbPassword },
  wsProtocols: ["binary"],
});
```

`wsUrl` is whatever the API returned: `"/ws/vnc/" + session_id`. But the gateway endpoint requires authentication via a JWT query parameter:

```go
// gateway/internal/httpapi/vnc_handler.go:289-301
token := r.URL.Query().Get("token")
if token == "" { http.Error(w, "token required", 401); return }
claims, err := deps.JWT.VerifyToken(token); ...
```

Neither the API response nor the web client appends `?token=<jwt>`. The upgrade will return 401 before the RFB handshake even starts.

**Fix:** the API can either issue a short-lived noVNC-scoped token and embed it into `ws_url`, or web can append the user's access token client-side. Spec §9 says the browser uses the user's JWT — so client-side appending is consistent with the spec.

### 2.4 Significant gap V3 — `wsUrl` is path-only, not absolute

The handler returns `"/ws/vnc/{sid}"`, which the browser passes verbatim to `new RFB(...)`. noVNC requires an absolute `ws://` or `wss://` URL. The frontend never prepends scheme + host. Combined with V2 this is currently a non-issue (it fails before that), but it must be fixed alongside.

### 2.5 Minor gap V4 — `relay.pump` writes WebSocket frames as raw bytes to `UnderlyingConn()`

```go
// gateway/internal/vncrelay/relay.go:209-217
msgType, reader, err := src.NextReader()
...
n, err := io.CopyBuffer(dst.UnderlyingConn(), reader, msgBuf)
```

`UnderlyingConn()` returns the *raw TCP connection*, bypassing the gorilla/websocket framing on the destination side. The peer expects WebSocket binary frames; what it receives will be unframed bytes. This breaks the byte relay in both directions.

The correct call is `dst.NextWriter(websocket.BinaryMessage)` followed by `io.CopyBuffer(writer, reader, ...)` and `writer.Close()`.

This is a real bug, not a path/wiring issue: even if V1, V2, V3 are fixed, the byte stream is corrupted before it ever reaches the agent.

### 2.6 Minor gap V5 — `GET /api/v1/jobs/{id}/vnc` ignores the live relay

The "is there an existing session?" handler (`vnc_handler.go:233-269`) reads from `deps.VNC` (DB store) but never from `deps.VNCRelay` (in-memory relay where the actual `RFBPassword` lives). On a page reload mid-session the user can see a session exists but cannot reconnect to it because the password isn't surfaced. This is a UX gap that compounds V1.

## 3. Login support (auto-open VNC, save login, re-inject)

### 3.1 Critical gap L1 — No auto-trigger when the agent hits a login wall

The user's question explicitly asked about this. There is no mechanism anywhere in the codebase that:

1. Lets the agent signal "I need a human to log in".
2. Lets the device propagate that to the gateway.
3. Lets the gateway notify the web UI to pop the VNC viewer (or auto-open one).

Search results:

- `agent/iagent_agent/server.py` exposes `/vnc/start` and `/vnc/stop` only — no event for "login_required".
- `agent/iagent_agent/runtime/executor.py` does not emit any login-related progress event.
- `device/iagent_device/jobs/dispatcher.py` polls only `status, percent, message` — no event types.
- `gateway/internal/model/types.go` defines no `WSEventLoginRequired`, `JOB_LOGIN_REQUIRED` frame, or similar.
- `web/src/pages/JobsPage.tsx` only shows the "Open Browser" button when `job.status === "running"` (`JobsPage.tsx:292-297`); it never auto-opens.

The spec also doesn't define this event yet — `docs/spec/05-tunnel-protocol.md §4.3` only has `JOB_PROGRESS / JOB_RESULT / JOB_REJECTED`. So this is both a **spec gap** and an **implementation gap**.

The agent-side runbook (`agent/AGENTS.md`) describes a manual QR/login flow where the agent saves a screenshot of the login page to `/work/output/login_qr.png` and waits 120 s for redirect — but the user has no signal that the screenshot is ready, no way to view it in the web UI in real time, and no nudge to open VNC.

**Recommended design (not present today):**

| Hop | Frame / Event | Payload | Behaviour |
|---|---|---|---|
| Agent → device | `POST /jobs/{id}/events` body `{type:"login_required", origin, label?}` | new endpoint on the agent runtime API | Agent calls this when it detects redirect-to-login or similar |
| Device → gateway | `JOB_LOGIN_REQUIRED` | `{job_id, origin, suggested_label?}` | New tunnel frame in §4.3 |
| Gateway | `WSEventLoginRequired` published on `pubsub.JobTopic(jobID)` | `{job_id, origin}` | DB row optional |
| Web UI | Shows toast "Login needed for {origin} — Open Browser" + auto-opens VNC dialog if user has it enabled |

Until that exists, the user's only option is to watch the progress message strings and click "Open Browser" manually after the job has reached `running` — and even that is broken today (V1, V2, V3, V4).

### 3.2 Critical gap L2 — `POST /vnc/{sid}/save-login` never collects the origin

```go
// gateway/internal/httpapi/vnc_handler.go:113-121
frame, _ := tunnel.NewFrame(model.FrameCredCapture, model.CredCapturePayload{
    SessionID: sessionID,
    JobID:     sess.JobID,
    AgentID:   sess.AgentID,
    Origin:    "",
    Label:     req.Label,
})
```

`SaveLoginRequest` (`web/src/components/VNCPanel.tsx:79-89` calls `apiClient.post('/vnc/{id}/save-login', { label })`) sends only `{ label }`. The handler hard-codes `Origin: ""`. The device then passes the empty string to the agent:

```python
# device/iagent_device/creds/relay.py:113-114
result = await client.get_browser_state(origin)  # origin = ""
```

The agent's `GET /browser/state?origin=` will return either the whole storage state or nothing depending on its filter logic; the captured cookies are then stored under an empty `origin` in `browser_credentials` and cannot be matched to a future job.

**Fix:** the web UI must let the user pick (or default to) the current page origin from the noVNC session. The gateway should require `origin` in `SaveLoginRequest` and validate it. The simplest implementation: agent exposes `GET /browser/url` that returns the current page URL, and the web UI fetches it before showing the "Save login" prompt — or better, asks the agent to capture the active tab's origin server-side.

### 3.3 Significant gap L3 — `CredCapturePayload` schema mismatch

Spec `05-tunnel-protocol.md §4.9`:

```
CRED_CAPTURE  D→G  { session_id, job_id, label, origin, storage_state, sha256 }
```

Device sends `data` (base64) instead of `storage_state`:

```python
# device/iagent_device/creds/relay.py:125-133
await self.outbox.enqueue_and_send(FrameType.CRED_CAPTURE, {
    "session_id": ..., "job_id": ..., "agent_id": ...,
    "label": ..., "origin": ...,
    "data": encoded,           # ← spec says storage_state
    "sha256": sha256,
})
```

Gateway reads `payload.Data`:

```go
// gateway/cmd/gateway/main.go:248
data, err := base64.StdEncoding.DecodeString(payload.Data)
```

So device + gateway agree, but **both diverge from the spec**. Either align the spec (fine; spec lists storage_state plain JSON, but gateway needs base64 anyway) or rename the Go/Python field to `storage_state`.

### 3.4 Minor gap L4 — `CRED_CAPTURE_ACK` direction flipped on errors

When the device fails to capture (agent unreachable etc.), it emits `CRED_CAPTURE_ACK` itself (`device/iagent_device/creds/relay.py:106-110, 135-138`). But the spec (§4.9) defines `CRED_CAPTURE_ACK` as **G→D only** (it's the gateway acknowledging that the capture was stored). The device should instead emit an `ERROR` frame or a tagged `CRED_CAPTURE` with empty state. As-is, the gateway's `OnCredCaptureAck` handler (`main.go:291-296`) gets surprise frames from the device and just logs them.

### 3.5 OK — `CRED_PUSH` (re-inject saved login on dispatch)

The forward path works:

| Step | Evidence |
|---|---|
| Allocator triggers `PushCredentialsForJob` after successful dispatch | `gateway/cmd/gateway/main.go:120-132`, `httpapi/jobs_handler.go:140-141` |
| Vault decrypts in-memory, frame `CRED_PUSH` sent | `httpapi/jobs_handler.go:292-318` |
| Device verifies sha256, calls agent `POST /browser/state`, ACKs `INJECTED` | `device/iagent_device/creds/relay.py:51-95` |
| Dispatcher waits for all `credential_ids` injected before `JOB_DISPATCH` | `dispatcher.py:85-86, 29-49` |

This is the one part of the login subsystem that is genuinely working.

## 4. Cross-cutting issues that affect the return path

### 4.1 Outbox `send_fn` signature mismatch (still open from audit 03)

Already documented as B1 in audit 03 (`docs/audit/03-local-device.md:170`). Every `JOB_RESULT`, `JOB_PROGRESS`, `FILE_PULL_*`, `VNC_OPENED`, `CRED_CAPTURE` goes through `Outbox.enqueue_and_send`, so if this is genuinely broken at runtime then the entire return path fails. The user reports the cloud-side dispatch already works, which suggests either the bug is latent or has been fixed since — please confirm. Recommend re-running audit 03's recommendation #1 before any of the above fixes are merged.

### 4.2 Web UI relies on schemas the server doesn't fulfil

Three places in `web/src` read fields the gateway does not return:

- `data.rfb_password` (V1)
- `wsUrl` as absolute URL (V3)
- (potentially) output file lists once C3 is addressed

Adding zod schemas with strict-mode parsing would have caught V1 immediately. Currently `useCredentials.ts:53` uses `apiClient.post<T>` with an inline interface that bypasses the `VNCOpenResponseSchema` registered in `web/src/api/schemas.ts:237`. Recommend funnelling all responses through the schema layer.

## 6. Recommendations (ordered by user impact)

### Tier 1 — unblocks "I want to see my result and use VNC" (this audit's core question)

1. **Fix workspace path mapping (C1).** Bind-mount `{data_dir}/work` to the agent's `/work` (or scope to `/work/workspaces` and `/work/output`), so the puller can actually read the agent's outputs. Without this nothing else in the file path matters.
2. **Implement `GET /api/v1/jobs/{id}/output` (list) and `GET /api/v1/jobs/{id}/output/{file_id}` (download) (C2).** Insert into `job_files` when receiving `FILE_PULL_END`, set `UserID` from the job. Tenant-scoped like other job routes.
3. **Surface output files in the web UI (C3).** Add a `JobOutputs` panel below the result JSON; one button per file using the new download endpoint.
4. **Fix VNC auth + password delivery (V1, V2, V3).** Recommended split:
   - POST `/jobs/{id}/vnc` blocks up to 15 s for `MarkReady`, then returns `{ session_id, ws_url: "wss://host/ws/vnc/{id}?token=<jwt-or-session-token>", rfb_password, ttl_secs }`.
   - Web client uses the absolute URL verbatim.
   - Or, asynchronous variant: subscribe to `vnc.ready` on the existing `/ws` event bus.
5. **Fix VNC relay framing bug (V4).** Use `NextWriter(BinaryMessage)` instead of `UnderlyingConn()`. Add a relay test that round-trips a known byte stream.

### Tier 2 — unblocks "agent reports it needs login" (the user's specific feature ask)

6. **Define a `JOB_LOGIN_REQUIRED` event chain (L1).** Smallest viable design:
   - Agent runtime API: `POST /jobs/{id}/events {type, origin, ...}` — already partially modeled by the spec's "events" idea (`docs/spec/04-agent-container.md`).
   - New tunnel frame `JOB_LOGIN_REQUIRED` in `05 §4.3`.
   - Gateway: handle, persist, publish `WSEventLoginRequired`.
   - Web UI: toast + optional auto-open of VNCPanel.
7. **Add `origin` to `save-login` (L2).** Web UI captures the active page origin (extra agent endpoint `GET /browser/url` or pull from VNC URL bar). Validate non-empty in the handler.
8. **Reconcile `CRED_CAPTURE` field name (L3).** Either update the spec to call it `data` (with note that it is base64-encoded storage_state) or rename device + gateway code to use `storage_state`.

### Tier 3 — robustness / polish

9. Implement `PushFilesToDevice` (C4) — without it inputs never reach agent.
10. Pull outputs on `failed` and `cancelled` jobs (C5).
11. Wire backpressure into the puller (C6) — wait for `FILE_PULL_ACK`, retry on `ERROR`.
12. Re-check Outbox signature mismatch (audit 03 B1) before/after merging the above.
13. Funnel all REST responses through zod schemas in `web/` to catch field-mismatch regressions early.
14. `GET /jobs/{id}/vnc` should consult the in-memory relay (V5) so reload after disconnect works.
15. Fix `CRED_CAPTURE_ACK` direction (L4) — device should emit `ERROR` instead.

## 7. Testing checklist (post-fix)

A production-grade end-to-end test for the return + interactive paths:

1. Submit a job whose command writes 1 file to `/work/output/result.txt` and another to `/work/output/sub/x.json`. Verify both files appear under `GET /jobs/{id}/output` and download with correct sha256.
2. Submit a job that calls `requests.get("https://protected.example.com")` and observe `JOB_LOGIN_REQUIRED` propagate to the web client.
3. Click "Open Browser" — verify noVNC connects, password works, mouse and keyboard round-trip.
4. Click "Save login" with a real origin; submit a follow-up job referencing the new `credential_id`; verify the agent starts already authenticated.
5. Kill the device tunnel mid-pull and reconnect — verify the puller resumes (currently no retry; gap C6) or at minimum the partial file is cleaned up.
